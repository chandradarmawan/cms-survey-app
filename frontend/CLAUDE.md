# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Pelindo Survey CMS — admin front-end for managing customer-satisfaction (SKP) questionnaires. **No backend**: all data lives in an in-memory Zustand store seeded at startup. **`docs/PRD.md` is the source of truth** — code comments reference its sections (`PRD §6.3`, `§8.4`, etc.); read the relevant section before changing behavior. UI labels are Indonesian (`src/i18n/id.ts`); identifiers/comments are mixed Indonesian/English.

## Commands

```bash
npm run dev        # vite dev server, http://localhost:5173
npm run build      # tsc -b (typecheck) + vite build
npm run typecheck  # tsc -b --noEmit — run this to validate changes
npm run preview    # serve the production build
```

No test runner and no linter are configured. `npm run typecheck` is the only automated check.

## Architecture

**Single store, two layers.** `src/store/useSurveyStore.ts` is the entire "database" (Zustand), initialized from `src/store/seed.ts`. `src/data/*.ts` is a thin access layer whose function signatures mirror a REST API (each is commented with its intended `GET/POST/PATCH` route) — this is the seam for swapping to a real `fetch` backend later. Pure read helpers (`filterSurveys`, `computeSummary`) take the data array as an argument; mutations call `useSurveyStore.getState().<action>()`. Components read state with selectors and mutate only through actions, so the UI stays in sync without refetching.

**Denormalized counts are an invariant maintained inside actions.** `Survey.jumlahPertanyaan` and `Question.childCount` are stored, not derived at render time. Any action that adds/removes/moves a question must recompute them (see `addQuestion`/`deleteQuestion` calling `countNonGroup`/`recalcChildCount`). Preserve this when editing store actions.

**Questions are a flat array with a `parentId` tree.** Hierarchy (groups → sub-questions) is modeled by `parentId`; the tree is assembled in the UI (`QuestionTree.tsx`). `deleteQuestion` cascades to all descendants. Question `kode` ("C1", "C3.1") is auto-generated by `nextKode` in `src/lib/questionMeta.ts` — top-level is `C{n}`, children are `{parentKode}.{m}`.

**Question types drive everything.** The 8 `QuestionType`s in `src/types/index.ts` determine which editor fields, badges, and scales apply. `src/lib/questionMeta.ts` centralizes per-type metadata (label/icon, tile order, tree badge) and the type predicates (`isPilihan`, `isSkala`, `scaleTypeFor`); reuse these rather than re-switching on `tipe`.

**Routing & navigation.** `src/routes.tsx` nests all pages under `AppShell` (top tabs, not a sidebar). Per-survey pages live at `surveys/:surveyId/{questions,scales,results}`. Features are organized by folder under `src/features/*` (survey-list, survey-create, question-manage, question-editor, scales, master-data, results).

**Conventions.** `@/` aliases `src/` (configured in both `vite.config.ts` and `tsconfig`). IDs are generated by `genId(prefix)` in `src/lib/id.ts` (`srv`/`q`/`idf`/…). Tailwind tokens (flat, light-mode, no shadows except modals) are defined in `tailwind.config.js` per PRD §4.
